SAAM 软件架构分析方法

SAAM是最早形成文档并广泛应用的架构分析方法，主要分析可修改性

📖 基本信息

全称：Software Architecture Analysis Method

提出：卡耐基梅隆大学SEI · Kazman等人 · 1983年

特点：最早形成文档并广泛应用的软件架构分析方法

主要分析内容：可修改性

评估技术：场景技术

架构描述：围绕功能、结构和分配描述

知识库可复用性：不涉及

// SAAM —— 最早形成文档的架构分析方法 (Kazman, 1983)

输入: [问题描述, 需求说明, 架构描述]

function SAAM分析(架构, 场景集) {
    // 第1步: 场景开发 —— 构造测试场景
    场景集 = 开发场景(需求说明)

    // 第2步: 架构描述 —— 功能/结构/分配
    架构描述 = 描述架构(架构, {功能, 结构, 分配})

    // 第3步: 单个场景评估 —— 逐一验证
    for (场景 of 场景集) {
        评估场景(架构描述, 场景)  // 检查能否支持
    }

    // 第4步: 场景交互 —— 发现场景间冲突
    分析交互(场景集)  // 多个场景是否矛盾

    // 第5步: 总体评估 —— 汇总结论
    return 总体评估(结果)  // 关注: 可修改性
}

// 核心: 通过场景技术评估架构的可修改性
// 知识库: 不可复用（每次分析独立构建）
核心：SAAM通过场景技术评估架构，用构造的场景测试架构是否满足需求。 主要关注可修改性，即架构能否方便地进行修改。分析过程五步： 场景开发 → 架构描述 → 单个评估 → 场景交互 → 总体评估。
📡 空中交通管制系统 🎵 嵌入式音频系统 🔧 修正控制系统 🔍 场景驱动评估 📊 聚焦可修改性

✅ 优点

  • 最早形成文档的架构分析方法，成熟度高
  • 可用于评价单系统或进行多系统比较
  • 场景驱动，直观易懂

❌ 缺点

  • 主要关注可修改性，分析维度有限
  • 知识库不可复用，每次需重新构建
  • 场景开发依赖分析人员经验
