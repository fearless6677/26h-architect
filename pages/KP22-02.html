Lambda架构三层结构与实现

Lambda架构分三层——批处理层(存储主数据集+批处理)、加速层(实时处理)、服务层(合并视图响应请求)，容错好但编码量大

批处理层 (Batch Layer)
存储主数据集(原始、不可变、真实)，周期执行批处理 → 生成批视图
HDFS / HBase + Spark / MapReduce

服务层 (Serving Layer)
响应用户请求，合并批视图 + 实时视图 → 最终结果
HBase / Cassandra + Hive 
// Lambda架构 —— 三层协作处理大数据

// ===== 1. 批处理层 Batch Layer =====
// 主数据集: 原始数据, 不可变, 真实
master_data = store_on_hdfs(raw_data)     // HDFS存储原始数据
batch_view  = spark_batch(master_data)     // 周期性全量批处理
write_to(batch_view, "HBase")            // 结果写入HBase

// ===== 2. 加速层 Speed Layer =====
// 只处理增量数据, 速度优先
realtime_view = spark_streaming(new_data)  // 实时流处理增量
write_to(realtime_view, "HBase")         // 实时视图写入

// ===== 3. 服务层 Serving Layer =====
// 合并批视图 + 实时视图, 响应查询
function query(user_request) {
    result = merge(batch_view, realtime_view)  // 合并两个视图
    return respond(result)                     // 返回给用户
}

// 关键: 批处理保证正确性, 加速层保证实时性, 服务层合并两者

核心：Lambda架构分三层—— 批处理层存主数据集、周期全量处理保证正确性； 加速层处理增量数据保证实时性； 服务层合并两个视图响应请求。 容错好，但要维护两套处理逻辑，编码量大。
🎬 视频网络平台：离线Spark/MR + 实时Streaming → HBase 📢 广告平台：Hive→MySQL + Spark Streaming→Redis + Java Web 📊 大数据分析：HDFS批处理 + Storm实时 + Cassandra服务

✅ 优点

  • 容错性好——主数据集不可变，可随时重算
  • 查询灵活度高——批视图+实时视图按需合并
  • 弹性伸缩，易于扩展
  • 历史数据批式全量处理，吞吐量大

❌ 缺点

  • 编码量大——需实现两套处理逻辑
  • 持续处理成本高——两套系统持续运行
  • 重新部署和迁移成本高
  • 实时与离线处理结果可能不一致
  • 需维护两套系统，复杂度高
