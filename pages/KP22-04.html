// ===== Lambda 架构：两套系统 =====

// 批处理层 —— 周期性全量处理，吞吐大，历史数据强
batch runBatchJob(historyData) {
    result = spark.read("hdfs://all-data")
             .filter(...).aggregate(...)
    write batchView(result)    // 产出批视图
}

// 实时层 —— 持续增量处理
stream runStreamJob(realtimeStream) {
    result = flink.from(realtimeStream).window(5min).aggregate(...)
    write realtimeView(result) // 产出实时视图
}

// 服务层：合并批视图 + 实时视图
serve = merge(batchView, realtimeView)

// ⚠️ 批视图与实时视图可能冲突！两套代码逻辑可能不一致


// ===== Kappa 架构：一套系统 =====

// 只有流处理，代码统一
stream runStreamJob(logTopic, fromOffset) {
    result = flink.from(logTopic, from=BEGINNING).aggregate(...)
    write servingLayer(result)  // 唯一视图，无冲突
}

// 需要重算？从头重放消息日志即可
replay(kafkaLog, from=offset_0)   // 全量重放 = 批处理效果

// ✅ 一套代码，逻辑永远一致
// ⚠️ 全量重放时吞吐量较低，海量历史数据处理能力弱

🧭 选型决策表：4大因素

决策因素→ 选 Lambda→ 选 Kappa
业务需求依赖 Hadoop / Spark / Storm 生态依赖 Flink 偏流式计算
历史数据频繁使用海量历史数据仅小规模数据集
成本预算成本充足预算有限
算法迭代算法稳定，不需要频繁修改频繁修改算法 / 模型参数
决策口诀：
📊 海量历史 + 成本充足 → Lambda
💰 小规模 + 成本有限 → Kappa
⚡ Flink流式 + 算法常变 → Kappa
🔧 Hadoop/Spark + 双引擎 → Lambda
核心：Lambda = 两套系统（批+流），复杂度高、成本高，但历史数据处理能力强、吞吐量大； Kappa = 一套系统（统一流式），复杂度低、成本低，但历史数据处理弱、吞吐量有限。 选型取决于业务需求、复杂度、成本、历史数据需求四大因素。
🏗️ Lambda：Hadoop + Spark + Storm 🌊 Kappa：Flink 单一引擎 📊 海量历史数据 → Lambda 💰 成本有限 → Kappa 🔄 算法频繁变更 → Kappa

✅ 各自优势

  • Lambda：历史数据处理能力强、吞吐量大
  • Kappa：代码统一维护简单、成本低

❌ 各自劣势

  • Lambda：需维护两套系统，开发维护成本高
  • Kappa：吞吐量低，历史数据处理能力弱
