Software Architectural Structures 软件架构结构三大分类

软件架构结构三大类：模块结构（代码/数据单元）、构件与连接器结构（运行时行为和交互）、分配结构（系统与非软件结构的关联）。

🖥️

分配结构

Allocation Structures
将决策体现为系统如何关联到环境中的非软件结构（CPU、网络、团队等）
部署 · 环境
模块(静态代码) → 构件与连接器(运行时交互) → 分配(部署到硬件/团队) 
// 软件架构结构三大分类 —— 不同维度看系统

// ① Module Structures — 静态代码层面
class ModuleStructure {
    modules = ["AuthModule", "PaymentModule", "DataAccessLayer"]
    // 决策：哪些代码/数据单元需要构建或采购？
    // 关键词：静态结构、代码组织、依赖关系
}

// ② Component and Connector Structures — 动态运行时层面
class CCStructure {
    components = ["WebServer", "Database", "MessageQueue"]
    connectors = ["HTTP", "TCP", "RPC"]
    // 决策：系统如何结构化为一组有运行时行为和交互的元素？
    // 关键词：运行时行为、交互、数据流
}

// ③ Allocation Structures — 部署与环境层面
class AllocationStructure {
    mapping = {
        "WebServer"   → "CPU-A, 192.168.1.10",
        "Database"    → "File-System / SSD",
        "AuthModule"  → "Team-Alpha (开发团队)"
    }
    // 决策：系统如何关联到非软件结构？
    // 关键词：CPU、文件系统、网络、开发团队
}

// 系统架构 = 功能→软硬件构件的映射 + 软件→硬件映射 + 人机交互

💡 核心理解

系统架构是功能到软硬件构件的映射、软件架构到硬件架构的映射，以及人机交互的关注。三类结构分别从静态代码、动态运行时、部署环境三个维度看同一个系统。

记忆口诀：模块=代码、构件=运行时、分配=环境。 三类结构是不同维度看同一个系统——静态怎么拆代码？运行时怎么交互？部署到哪台机器/哪个团队？
📦 Module → 代码或数据单元 🔌 C&C → 运行时行为和交互 🖥️ Allocation → 非软件结构（CPU/网络/团队）

✅ 优点

  • 分类清晰，三类对应不同维度
  • 练习题有详细解析，复习效率高
  • 覆盖静态/动态/环境全方位视角

❌ 难点

  • 三类结构的英文名称长且容易混淆
  • 需要准确区分每类结构的含义
  • Component和Module概念易混
