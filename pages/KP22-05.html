核心差异：Λ 维护两套代码（批+流），Κ 只维护一套流代码，重算从 Kafka 重消费 

// ━━━━━━ Λ Lambda 架构 ━━━━━━ 批处理 + 实时流 双路写存储

// 批处理链路（准确但慢 — 处理全量历史数据）
batchLayer = Spark/MR
  .read("HDFS")              // 全量历史数据
  .aggregate(daily_summary)   // 离线聚合
  .write("HBase")            // 写入存储

// 实时加速链路（快但近似 — 处理增量数据）
speedLayer = SparkStreaming
  .read("Kafka")              // 增量数据
  .window(5_min)             // 滑动窗口
  .write("HBase")            // 同一存储

// 服务层：合并两路结果
query(user_id) → merge(batch_view, stream_view)

// ━━━━━━ Κ Kappa 架构 ━━━━━━ 一切皆流，Flink 统一处理

streamOnly = Flink
  .from("Kafka")              // 数据入口
  .window(tumbling: 1h)      // 用窗口模拟批处理
  .aggregate(result)          // 统一计算
  .sink("ElasticSearch")      // 写入存储

// 💡 需要重算？从 Kafka 重新消费即可！
// 💡 关键区别：Kappa 只维护一套流处理代码

// ━━━━━━ 考试答题要点 ━━━━━━
// ① 识别架构类型 → Λ 还是 Κ
// ② 说明选择理由 → 业务需要精确批处理？团队能否驾驭纯流？
// ③ 列举各层组件 → 采集/缓冲/计算/存储/服务
核心对比： Lambda(Λ) = 批处理 + 实时流双路并行，准确但维护两套代码； Kappa(Κ) = 一切皆流，只维护一套代码，重算从 Kafka 重消费。 选 Λ 还是 Κ，关键看：是否需要精确批处理结果？团队能否驾驭纯流式？
