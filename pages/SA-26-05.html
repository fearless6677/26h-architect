📄 摘要

  • ❌ 项目背景缺失，读者不知道在做什么
  • ❌ 理论不足，没有点出用了什么理论/方法
  • ❌ 草草收尾，总结句敷衍
  • ❌ 字数不够（一般 300～500 字）
  • ❌ 分段（摘要应一整段，不拆段）

🏗️ 项目背景

  • ❌ 正文不介绍项目，直接开写理论
  • ❌ 非软件项目（如硬件、网络工程）
  • ❌ 项目太陈旧（超过 3 年）
  • ❌ 虚构项目，经不起追问
  • ❌ 技术过旧，没有亮点
  • ❌ "帽子太多"，堆砌无关背景
  • ❌ 背景与所用理论不匹配

🔗 过渡

  • ❌ 缺少过渡句，章节之间突兀跳跃
  • ❌ 过渡生硬，套模板痕迹明显

📚 理论

  • ❌ 超 700 字，大段抄书喧宾夺主
  • ❌ 理论缺失，全文看不出用了什么方法
  • ❌ 未响应题干，题干要求 A 理论却写了 B
  • ❌ 理论与实践混杂，分不清哪段在说什么
  • ❌ 概念不清，术语用错
  • ❌ 简单罗列，缺少与项目的关联分析
  • ❌ 论点未讲完就换话题

⚙️ 实践

  • ❌ 篇幅太短，实践才是论文核心
  • ❌ 纯理论空谈，没有具体做法
  • ❌ 知识点错误，暴露基础不牢
  • ❌ 未围绕论点，写到哪算哪
  • ❌ 只 Why 不 How——只说为什么，不说怎么做
  • ❌ 过于微观，陷入代码/配置细节
  • ❌ 技术不当，用了不该用的工具

🏁 结尾

  • ❌ 未呼应论点，首尾脱节
  • ❌ 问题滔滔不绝，写成"检讨书"
  • ❌ 单薄或冗长，把握不好度
  • ❌ 前后矛盾，与摘要或正文冲突
// 论文自查清单 —— 提交前逐段扫描
function reviewPaper(paper) {
    const issues = [];

    // 1. 摘要
    if (paper.abstract.hasParagraphs())  issues.push("摘要不能分段！");
    if (paper.abstract.words < 300)      issues.push("摘要字数不够");
    if (!paper.abstract.hasTheory())      issues.push("摘要缺少理论/方法");

    // 2. 项目背景
    if (paper.background.isTooOld())      issues.push("项目太陈旧");
    if (!paper.background.isSoftware())   issues.push("必须写软件项目");

    // 3. 过渡
    if (paper.transitions.count === 0)    issues.push("缺少过渡句");

    // 4. 理论
    if (paper.theory.words > 700)         issues.push("理论超 700 字！");
    if (!paper.theory.matchPrompt())       issues.push("理论未响应题干");

    // 5. 实践 —— 最常扣分
    if (paper.practice.onlyWhyNoHow())    issues.push("只 Why 不 How！");
    if (paper.practice.tooShort())        issues.push("实践篇幅太短");

    // 6. 结尾
    if (paper.conclusion.conflictsWith(paper.abstract))
        issues.push("前后矛盾！");

    return issues;
}
// 关键：提交前跑一遍 reviewPaper()，所有 warn 清零再交！
核心：论文六大部分每段都有常见扣分陷阱。 摘要不能分段、理论不能超 700 字、实践必须写 How 不能只写 Why、结尾必须呼应论点、前后一致。 写完论文后对照清单逐条自查，是性价比最高的提分手段。
📄 摘要不分段 📚 理论 ≤ 700 字 ⚙️ 必须写 How 🏁 首尾呼应 🔗 过渡要自然

✅ 使用建议

  • 对照清单自查可有效提分
  • 写完论文逐部分扫描，消灭低级错误
  • 重点关注高频扣分项：摘要分段、实践只 Why 不 How
