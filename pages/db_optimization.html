── 反规范化设计 ──────────────────────────

策略1: 增加冗余列（避免 JOIN）
  订单表 += 客户姓名列
  // SELECT 不再需要 JOIN 客户表

策略2: 增加派生列（减少计算）
  订单表 += 订单总额列 (= 单价 × 数量)
  // SELECT 不再每次计算

策略3: 重新组表 — 调整表结构关系
策略4: 水平分割 — 按行拆分（如按用户ID取模）
策略5: 垂直分割 — 按列拆分（大字段独立成表）

── 分布式优化 ───────────────────────────

1. 主从复制:
   主库写 → binlog → 从库 relaylog → 回放
   模式: 全同步 | 半同步 | 异步

2. 读写分离: 主写从读，提升并发
3. 分表: 水平切分(大表拆小) | 垂直切分(列分组)
4. 分库: 业务隔离，独立实例
冗余列 订单表加"客户姓名"列 — 避免每次查询订单时都 JOIN 客户表，用空间换时间。
派生列 订单表加"订单总额"列 — 由 单价×数量 预计算得出，避免每次查询都做乘法运算。
反规范化缺点 修改客户姓名时 — 需要同步更新所有订单表中的冗余"客户姓名"字段，否则数据不一致。
主从复制 MySQL 主从流程 — 主库写 binlog → 从库读取到 relaylog → 从库回放执行，实现数据同步。
水平分割 用户表按 ID 取模分4张表 — user_0, user_1, user_2, user_3，每张表数据量降为1/4。
分库 电商拆分用户库/订单库/商品库 — 不同业务独立数据库实例，互不影响。
核心：数据库优化分两个层面。 集中式优化采用反规范化设计——增加冗余列(避免连接操作)、增加派生列(减少计算量)、重新组表、水平分割表(按行拆分)、垂直分割表(按列拆分)，优点是提高查询性能，缺点是数据冗余和一致性问题。 分布式优化采用主从复制(读写分离基础)、读写分离(主写从读提升并发)、分表(大表拆小表)、分库(不同业务独立实例)。MySQL主从通过binlog同步，有语句级、行级、混合三种复制模式。

✅ 优点

  • 查询性能显著提升
  • 支持更大并发量
  • 降低单库压力

❌ 缺点

  • 数据冗余增加
  • 一致性维护复杂
  • 存储空间浪费

📝 考试关键词

反规范化 冗余列 派生列 水平分割 垂直分割 主从复制 读写分离 分表分库 binlog 全同步 半同步 异步

🔗 关联知识

  • 分布式数据库(distributed_db) = 分库分库的底层架构支撑
  • Redis/NoSQL(nosql_redis) = 数据库优化的另一条路径——NoSQL bypass
  • MySQL复制模式：语句级(STATEMENT) / 行级(ROW) / 混合(MIXED)
  • 反规范化 vs 规范化：规范化减冗余保一致，反规范化加冗余换性能
  • CAP定理：分布式优化必须在一致性与可用性之间取舍
