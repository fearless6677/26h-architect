Kappa架构结构与特点

Kappa架构删除Lambda的批处理层，仅保留实时层(流式处理)和服务层，统一离线与实时代码，维护简单但吞吐量较低

⚖️ Lambda架构 vs Kappa架构

维度Lambda（三层）Kappa（两层）
批处理层✅ 有（MapReduce/Spark批处理）❌ 删除，由流式引擎重放历史代替
数据通道分布式文件系统消息队列Kafka替代
服务层✅ 有（统一视图）✅ 有（数据湖存储）
代码维护⚠️ 两套代码（批+流）✅ 一套代码统一处理
复杂度较高（三层协调）较低（两层简化）
吞吐量高（批处理大批量）相对较低（流式逐条处理）
历史数据强（批处理擅长）弱（需从Kafka重放，受保留期限制）
本质：改进Lambda的加速层，使其既能实时处理又能重处理历史数据。 常见变形：Kappa+架构、混合分析系统Kappa架构。 
// Kappa架构 —— 删除批处理层，一切皆流

// ① 数据源 → Kafka 消息队列（替代数据通道）
Producer → Kafka Topic("raw-events") → 持久化原始数据

// ② 实时层：Flink 消费 Kafka，逐条流式处理
Flink job {
  source(Kafka)              // 从Kafka读数据流
    .filter(e => e.isValid)   // 清洗过滤
    .window(Tumbling(5min))  // 滚动窗口聚合
    .sink(DataLake("s3://"))  // 写入数据湖
}

// ③ 服务层：从数据湖读取实时视图
API → query(DataLake) → 返回结果给用户

// 💡 历史数据重处理？
//    不需要批处理！直接重启Flink job从Kafka偏移量重放
//    同一段代码，换个offset就搞定 ✅

// 与Lambda对比：
//   Lambda = 实时层 + 批处理层 + 服务层（两套代码）
//   Kappa  = 实时层 + 服务层（一套代码）
核心：Kappa架构在Lambda基础上删除批处理层，以消息队列(Kafka)替代数据通道，只保留两层： (1)实时层——采用流式处理引擎(Flink/Spark Streaming)逐条处理输入数据生成实时视图； (2)服务层——使用实时视图结果响应用户请求，实践用数据湖存储。 本质是改进Lambda的加速层，使其既能实时处理又能重处理历史数据。最大优点：离线和实时代码统一，维护方便。

✅ 优点

  • 离线和实时处理代码统一
  • 方便维护，一套代码搞定
  • 复杂度低，成本低
  • 计算开销相对较小

❌ 缺点

  • 消息中间件有性能瓶颈
  • 数据关联时处理开销大
  • 抛弃了离线计算的可靠性
  • 流式全量处理吞吐量相对较低
  • 历史数据处理能力相对较弱
