微服务设计约束

微服务四层设计约束：个体约束（业务独立）、横向关系（可发现性+可交互性）、纵向约束（数据存储隔离DSS）、分布式约束（CI/CD+蓝绿/金丝雀发布）。

// 微服务四层设计约束

// ① 个体约束 —— 每个服务业务独立
service OrderService {   // 只管"订单"领域
    api.createOrder()
    api.queryOrder()
}
service InventoryService { // 只管"库存"领域
    api.deductStock()
    api.queryStock()
}

// ② 横向约束 —— 可发现性
registry.register("OrderService", "10.0.0.1:8080")
registry.discover("InventoryService")  // 自动感知，无需重新发布

// ③ 纵向约束 —— DSS 数据存储隔离
// ✅ 正确：通过API访问
InventoryService.api.deductStock(order.sku)
// ❌ 错误：订单服务直接访问库存数据库！
// OrderService.query("SELECT * FROM inventory_db...")  ← 违反DSS

// ④ 分布式约束 —— 全自动化CI/CD
pipeline {
    build → test → canary("1%") → canary("20%") → fullRelease()
}
核心：微服务将单体应用拆为松耦合的多个子应用。 四层约束从个体→横向→纵向→全局层层递进： 每个服务业务独立；服务间可发现+可交互； 数据隔离（DSS）；全局CI/CD+灰度发布。
🔍 可发现性：服务注册中心自动感知 🔒 DSS：数据存储隔离 🔵🟢 蓝绿发布：新旧版本一键切换 🐤 金丝雀发布：灰度逐步放量 ⚙️ CI/CD：全自动化流水线

✅ 优点

  • 松耦合提高迭代效率
  • 独立部署和测试
  • 技术栈灵活

❌ 缺点

  • 分布式复杂性增加
  • 需要全自动化CI/CD支撑
  • 运维复杂度高

📝 考试关键词（2~4分 单选 + 案例题）

微服务 可发现性 可交互性 DSS 数据存储隔离 CI/CD 蓝绿发布 金丝雀发布

典型考点：服务扩容后依赖方如何感知？→ 可发现性（服务注册中心）。 订单服务能否直接读库存库？→ 不能，违反DSS原则。 先向1%用户发布验证？→ 金丝雀发布。
