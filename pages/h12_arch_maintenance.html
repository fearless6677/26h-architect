// 架构维护 —— 四方面 + 六指标

架构维护 {
  知识管理:
    架构知识 = 架构设计 + 设计决策
    // 在架构文档中捕捉知识，防止关键知识"沉没"

  修改管理:
    隔离区域 → 影响最小化
    // sandbox中测试变更后再上线

  版本管理:
    演化控制 + 使用 + 评价
    // 为版本演化提供依据
}

// 可维护性度量 6 指标
metrics = {
  CNN  → 圈复杂度     // 越低越好
  FFC  → 扇入扇出度   // 模块被调用/调用他者
  CBO  → 模块间耦合度 // 越低越好
  RFC  → 模块的响应   // 能触发的方法数
  TCC  → 紧内聚度     // 越高越好
  LCC  → 松内聚度     // 越高越好
}
核心：架构维护四大方面 —— (1)知识管理：架构知识 = 架构设计 + 架构设计决策，在文档中捕捉，防止"沉没"；(2)修改管理：建立隔离区域，保障修改影响最小；(3)版本管理：为版本演化控制、使用和评价提供依据；(4)可维护性度量：6个量化指标(CNN/FFC/CBO/RFC/TCC/LCC)。
📄 记录设计决策 → 知识管理 🧪 sandbox测试变更 → 修改管理 📊 圈复杂度15→8 → 可维护性度量 🏷️ 版本演化控制 → 版本管理

✅ 优点

  • 全面覆盖维护各维度
  • 6个量化指标便于度量
  • 知识沉淀防止"沉没"
  • 隔离区保障修改安全

❌ 缺点

  • 知识管理容易被忽视
  • 度量指标需持续跟踪
  • 隔离区增加维护成本
  • 版本管理流程复杂
