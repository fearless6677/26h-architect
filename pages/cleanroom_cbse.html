净室与构件化软件工程

净室靠数学验证而非测试来保证正确性；构件化就像搭积木，用现成模块拼装软件。

📖 核心解释

净室软件工程(CSE)强调在开发中建立正确性，理论基础是函数理论和抽样理论，用盒子结构规约建模，以正确性验证(而非测试)为主要纠错机制。基于构件的软件工程(CBSE)基于分布对象技术，强调通过可复用构件构建系统。构件须具备可组装、可部署、文档化、独立性、标准化5个特征。CBSE与传统开发不同：需要早期完整需求、根据可用构件修改需求、架构设计特别重要。构件组装有顺序/层次/叠加三种方式，接口不兼容时需编写适配器。

💻 伪代码

// 净室软件工程(CSE)
理论基础: 函数理论 + 抽样理论
核心: 正确性验证(而非测试)作为发现错误的主要机制
使用: 盒子结构规约进行分析和设计建模
缺点: 太理论化、忽视测试、传统弊端

// 基于构件的软件工程(CBSE)
构件特征: 可组装 + 可部署 + 文档化 + 独立性 + 标准化

CBSE过程:
1. 系统需求概览
2. 识别候选构件
3. 修改需求匹配构件
4. 体系结构设计(特别重要)
5. 构件定制与适配
6. 组装构件创建系统

// 构件组装方式
顺序组装 / 层次组装 / 叠加组装
// 接口不兼容: 参数不兼容、操作不兼容、操作不完备
// 解决: 编写适配器构件

📌 实际例子

  • 净室用于航空等安全关键系统，通过数学验证保证正确性
  • 构件模型：Web Services、EJB、.NET
  • CBSE场景：从构件库找到支付模块，修改后组装到新系统
  • 接口不兼容示例：新构件参数类型与旧构件不同，需写适配器转换

✅ 优点

  • 净室：生成质量非常高的软件
  • CBSE：提高软件复用率，降低开发成本
  • 构件化：缩短开发周期

❌ 缺点

  • 净室太理论化，忽视测试
  • CBSE需要早期完整需求
  • 构件接口不兼容需要适配器
  • 架构设计阶段特别重要
