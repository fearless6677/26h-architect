MVC：Controller 接收用户输入，调用 Model 处理业务逻辑，再更新 View 显示结果。View 可直接访问 Model（耦合较高）。
维度MVCMVPMVVM
中间层ControllerPresenterViewModel
View↔Model 通信可直连（耦合）必须经 Presenter必须经 ViewModel
数据绑定无手动同步双向自动绑定
可测试性一般Presenter 可脱离 UI 测试 ✅ViewModel 可脱离 UI 测试 ✅
复用性Controller 复用有限Presenter 可复用多个视图 ✅ViewModel 可复用 ✅
代表框架Struts2Android (传统)Vue.js / WPF
 
// ====== MVC ====== Struts2 风格
class UserAction {               // Controller (Action)
    private User user;           // Model (POJO)
    public execute() {
        user.save();              // 调用 Model
        return "success";        // → View (JSP)
    }
}

// ====== MVP ====== Android 风格
interface IView { showData(data); }
class Presenter {
    private IView view;          // 持有 View 接口引用
    private Model model;
    public loadData() {
        var data = model.fetch();
        view.showData(data);     // 通过接口更新 UI
    }
}

// ====== MVVM ====== Vue.js 风格
<template>
  <input v-model="username" />  // 双向绑定！
  <p>{{ greeting }}</p>
</template>
<script>
export default {
    data() { return { username: '' } },
    computed: {
        greeting() { return `Hello, ${this.username}`; }
    }
}
</script>
演进核心：从 MVC 的"View 可直连 Model" → MVP 的"View 与 Model 完全隔离，经 Presenter 中转" → MVVM 的"通过 DataBinding 双向绑定，View 与 Model 只能经 ViewModel 通信"。每一步都在追求更彻底的解耦。
🏛️ Struts2（MVC） 📱 Android Presenter（MVP） 💚 Vue.js 响应式（MVVM） 🪟 WPF / Silverlight（MVVM）

✅ 优点

  • 允许多种用户界面扩展
  • 易于维护
  • 增加应用可扩展性、强壮性、灵活性
  • MVP 支持 Presenter 复用和单元测试
  • MVVM 实现 View 与 Model 完全解耦

❌ 缺点 / 注意事项

  • MVC 中 View 可直接访问 Model，耦合较高
  • MVP 的 Presenter 逻辑可能过于臃肿
  • MVVM 的 DataBinding 实现复杂度高，调试困难
