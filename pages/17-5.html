服务网格（Service Mesh）与容器技术

服务网格将微服务间的连接、安全、流量控制和可观测下沉为平台基础设施，实现应用与基础设施解耦；容器将应用及依赖打包，不受环境限制。典型Mesh技术：Istio、Linkerd、Consul。

class ServiceA:
    def callB(req):
        # 应用代码: 像调用本地方法一样调用服务B
        return http.get("http://serviceB/api", req)
        # ↑ 实际请求被 Sidecar 拦截！应用无感知

class SidecarProxy:    // 每个服务实例旁边都有一个
    def intercept(req):
        service_discovery(req)   // 1. 服务发现：找到服务B地址
        rate_limit(req)          // 2. 限流：QPS 不超过阈值
        circuit_breaker(req)     // 3. 熔断：服务B故障时快速失败
        auth_mTLS(req)           // 4. 安全：mTLS 加密通信
        tracing(req)             // 5. 可观测：分布式追踪/指标
        return forward(req)      // 通过所有策略后，转发请求

// 控制平面 (Istio/Linkerd/Consul) 统一下发策略到各 Sidecar
// Sidecar = 数据平面(Data Plane)，负责实际流量转发
对比维度🐳 容器🖥️ 虚拟机📦 传统部署
隔离级别进程级（共享宿主机内核）硬件级（完整OS）无隔离
启动速度秒级 ✅分钟级 ❌手动部署 ❌
资源占用MB级 ✅GB级 ❌—
环境一致性打包全部依赖 ✅部分 ✅"在我机器上能跑" ❌
可移植性"一次构建，到处运行" ✅受限强绑定环境 ❌
密度（单机运行数）高 ✅低 ❌1个

容器编排（如 Kubernetes）提供以下核心能力：

📅 资源调度CPU/内存/节点分配
🚀 应用部署滚动更新 / 回滚
🔧 自动修复故障自愈、重启
🔍 服务发现自动注册与寻址
⚖️ 负载均衡流量均匀分配
📈 弹性伸缩HPA / VPA 自动扩缩
📝 声明式 APIYAML 描述期望状态
🔌 可扩展架构CRD / Operator 模式
🌐 可移植性跨云 / 混合云部署
核心：服务网格将连接、安全、流量控制、可观测从应用代码中剥离，下沉到 Sidecar 代理层。 应用只关心业务，网络策略由控制平面统一管理。容器把应用+依赖打包，不再受环境限制。
🔗 Istio（最流行） 🔗 Linkerd（轻量级） 🔗 Consul（+服务发现） 🐳 Docker 容器 ☸️ Kubernetes 编排 🏗️ Sidecar 模式

✅ 优点

  • 应用与基础设施解耦，业务代码更干净
  • 通用功能下沉到 Sidecar，减少重复开发
  • 统一的流量控制和安全策略（全网格一致）

❌ 缺点

  • 引入额外网络跳数（Sidecar 代理）增加延迟
  • Sidecar 管理复杂（每实例一个，资源开销大）
  • 学习曲线陡峭（Istio 概念多、配置复杂）
