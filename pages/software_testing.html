软件测试（Software Testing）

测试就是用各种方法找软件的漏洞，从单个模块测到整个系统，层层把关。

🎬 测试阶段
📊 分类体系
💻 伪代码

测试阶段：从小到大，层层把关

单元测试 程序员做 · 白盒为主
集成测试 模块组装 · 白盒+黑盒
系统测试 功能 / 性能 / 安全 / 健壮性
验收测试 用户确认满足需求
✅ 系统测试结束标志：需求覆盖率达标 + 缺陷全部归零

测试分类体系（三个维度）

📌 按执行状态

静态测试 (ST)
不运行代码，审查/走查/静态分析
动态测试 (DT)
运行代码，输入数据验证输出

📌 按内部结构

黑盒测试
不看代码，只看输入输出
白盒测试
看代码逻辑，测路径/分支
灰盒测试
介于两者之间

📌 按执行方式

人工测试 (MT)
人手动操作执行
自动化测试 (AT)
脚本/工具自动执行
// 测试分类体系
// 按执行状态: 静态测试(ST) / 动态测试(DT)
// 按内部结构: 黑盒测试 / 白盒测试 / 灰盒测试
// 按执行方式: 人工测试(MT) / 自动化测试(AT)

// 测试阶段(从小到大)
function testLifecycle(module) {
    单元测试(module)      // 程序员自己做，白盒为主
        ↓
    集成测试(modules)     // 模块组装验证，白盒+黑盒
        ↓
    系统测试(system)      // 验证符合SRS，功能/性能/安全/健壮性
        ↓
    验收测试(user)        // 确认满足用户需求
}

// 系统测试结束标志
if (需求覆盖率达标 && 缺陷全部归零) {
    "测试结束 ✓"
}
核心：软件测试是在规定时间和成本内尽量多发现漏洞的活动。 四个阶段从小到大：单元测试（模块级，程序员做，白盒）→ 集成测试（组装验证）→ 系统测试（验证符合SRS，包括功能/性能/健壮性/安全性）→ 验收测试（确认满足用户需求）。 系统测试结束标志：需求覆盖率达标且缺陷全部归零。
🔍 单元测试：程序员用代码审查、静态分析测试自己写的模块 🔗 集成测试：将多个模块组装后验证接口是否正确 🖥️ 系统测试：检查完整系统是否符合需求规格说明书 ✅ 验收测试：用户确认系统是否满足业务需求

✅ 优点

  • 多层次测试体系确保质量
  • 黑盒白盒结合覆盖面广
  • 自动化测试提高效率

❌ 缺点/局限

  • 测试不能保证发现所有漏洞
  • 测试成本和时间有限

📝 考试关键词

黑盒测试 白盒测试 单元测试 集成测试 系统测试 验收测试 静态测试 动态测试 需求覆盖率 缺陷归零

🔗 关联知识

  • 系统分析与设计 = 测试的依据来源于需求规格说明书（SRS）
  • 软件质量管理 = 测试是质量保证（QA）的核心手段
  • 软件过程模型 = V模型中每个开发阶段对应一个测试阶段
