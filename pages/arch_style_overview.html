核心：架构风格 = 词汇表（构件 + 连接件）+ 约束（组合规则）。 不同家族用不同的方式组织构件和连接件，就像不同的「建筑风格」——有平房、高楼、别墅……各有各的规则。
🌊

① 数据流风格

数据驱动，像流水线一样加工

  • 批处理 — 整块传递，前步完成才启动后步
  • 管道-过滤器 — 序贯处理，数据流连接
构件：过滤器（数据处理单元）
连接件：管道（数据流传输通道）
约束：前一步的输出 = 后一步的输入；数据单向流动
典型：Unix shell、编译器流水线、ETL
📞

② 调用/返回风格

函数调用，层层分解

  • 主程序/子程序 — 结构化分解
  • 面向对象 — 对象间方法调用
  • 层次型 — 相邻层接口调用
  • C/S — 二层/三层客户端-服务器
  • B/S — 浏览器-Web服务器-数据库
构件：模块 / 对象 / 服务
连接件：过程调用 / 方法调用 / RPC
约束：上层调下层，逐层传递（层次型）
典型：OSI七层模型、MVC、微服务
🗄️

③ 以数据为中心

中央数据结构 + 独立操作构件

  • 仓库(Repository) — 中央数据结构 + 操作构件
  • 黑板(Blackboard) — 推理 + 知识源 + 控制
仓库：中央数据结构活跃，构件独立被动（如数据库系统）
黑板：适合不确定问题，知识源独立推理，控制器协调（语音识别、模式识别）
区别：仓库=数据驱动；黑板=问题求解驱动
🖥️

④ 虚拟机风格

提供虚拟执行环境

  • 解释器 — 执行效率低，灵活性好
  • 规则系统 — 知识库 + 规则解释器
构件：解释引擎、状态机
连接件：状态传递、规则匹配
特点：用软件模拟硬件，灵活但慢
典型：JVM、Python解释器、专家系统
🔗

⑤ 独立构件风格

构件独立运行，松耦合通信

  • 进程通信 — 消息传递
  • 事件系统 — 触发/广播事件（发布-订阅）
构件：独立进程/组件
连接件：消息队列 / 事件通道
特点：构件完全独立，通过事件/消息隐式通信
典型：GUI事件驱动、Kafka、WebSocket
🏗️

⑥ C2 风格

构件通过连接件连接，构件间无直接连接

  • 核心规则 — 所有通信必须通过连接件
  • 拓扑 — 构件在上/在下，连接件在中间
构件：功能模块
连接件：事件/消息总线
约束：构件之间不能直接通信，必须经过连接件
特点：支持在线更换构件，可插拔性强
典型：IDE插件系统、IDEA/Eclipse

🐧 管道过滤器: Unix shell 管道 cat file | grep pattern | sort 📡 层次型: OSI七层模型, TCP/IP协议栈 🌐 B/S三层: 浏览器→Web服务器→数据库服务器 🎤 黑板: 语音识别、模式识别、信号处理 🖱️ 事件系统: GUI事件驱动、发布-订阅模式
实例详解：

Unix Shell 管道 — cat file | grep "error" | sort | uniq -c
每个命令是一个过滤器，| 是管道连接件。数据逐条流过，命令可自由组合。

OSI 七层 / TCP/IP — 每层只与相邻层交互。
应用层调传输层，传输层调网络层...层间通过接口连接。

B/S 三层架构 — 浏览器（展示层）→ Web服务器（业务逻辑层）→ 数据库服务器（数据层）。
典型的层次型 + C/S 组合风格。

语音识别系统 — 经典黑板架构。
声学特征→音素→词汇→语法→语义，各知识源独立推理，控制器协调全局。

GUI 事件驱动 — 点击按钮触发 onClick 事件，事件总线分发给所有注册的监听器。
构件完全独立，通过事件隐式通信。

✅ 架构风格的价值

  • 实现架构复用 — 成熟风格可直接套用
  • 提供设计指导 — 减少设计决策的随意性
  • 反映领域共性 — 同类系统用类似风格
  • 降低复杂度 — 有约束就有秩序

⚠️ 风格选择的代价

  • 选错风格 → 系统难以演化
  • 过度设计 → 约束太多丧失灵活性
  • 混合风格 → 约束冲突，维护困难
  • 风格不匹配领域 → 性能/可维护性差
