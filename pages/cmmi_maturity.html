CMM/CMMI成熟度模型

CMMI是评估软件开发团队'成熟度'的5级阶梯，从混乱到持续优化。

📖 核心解释

CMMI(软件能力成熟度模型集成)在CMM基础上发展而来，将软件过程改进分为5个等级：初始级→已管理级→已定义级→量化管理级→优化级。核心考点：量化管理级与已定义级的区别在于对过程性能的可预测性——量化管理级能用统计方法预测过程和产品质量。CMM是概念模型，框架刚性但解释和实现有弹性。

💻 伪代码

// CMMI五个成熟度等级(从低到高)
Level 1: 初始级     → 无序，靠个人能力
Level 2: 已管理级   → 项目级管理
Level 3: 已定义级   → 组织级标准过程
Level 4: 量化管理级 → 过程和产品质量可预测(关键区分点)
Level 5: 优化级     → 持续改进

// Level 4 vs Level 3 核心区别
Level 3: 有标准流程，但结果不完全可预测
Level 4: 用统计方法管理，过程性能可预测

📌 实际例子

  • 初始级：项目管理混乱，成功靠个人英雄
  • 已管理级：项目有计划和跟踪，但各项目管理方式不同
  • 已定义级：组织有统一的标准过程，所有项目都遵循
  • 量化管理级：用统计数据量化管理过程和产品质量

✅ 优点

  • 提供明确的过程改进路径
  • 帮助组织系统提升软件质量

❌ 缺点

  • 模型框架刚性，不能随意改变
  • 实施成本高，需要大量文档
