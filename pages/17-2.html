云原生分布式事务模式

五种模式：XA（2PC强一致性能差）、BASE（最终一致性）、TCC（Try-Confirm-Cancel业务侵入强）、SAGA（补偿事务回滚）、SEATA AT（自动回滚无代码）

XA模式（两阶段提交 2PC）
协调者先询问所有参与者是否可以提交（Prepare），再统一发Commit/Rollback指令。强一致性，但需两次网络交互，性能差。适用于银行转账等对一致性要求极高的场景。 
// ===== 1. XA / 2PC —— 两阶段提交 =====
// Phase 1: Prepare（询问）
coordinator.ask([服务A, 服务B, 服务C])  // "你们能提交吗？"
// Phase 2: Commit / Rollback（执行）
if all_ok  → coordinator.commit(all)
else       → coordinator.rollback(all)

// ===== 2. BASE —— 最终一致性 =====
服务A.do()  // 立刻返回，不等别的服务
async { 消息队列通知 服务B → 服务B最终完成 }  // 短暂不一致，最终一致

// ===== 3. TCC —— Try-Confirm-Cancel =====
Try()     // 预留资源（冻结库存）
Confirm() // 确认提交（扣减库存）
Cancel()  // 取消回滚（释放冻结）

// ===== 4. SAGA —— 补偿事务 =====
T1 → T2 → T3 // 正向事务链
if T3 失败 → C2() → C1()  // 反向补偿链（倒序回滚）

// ===== 5. SEATA AT —— 自动回滚 =====
// 一阶段：业务SQL + 框架自动记录前后快照
// 二阶段：成功→异步删除快照；失败→框架自动用快照回滚
@GlobalTransactional  // 加一个注解就够了！零代码侵入
doBusiness()
核心：业务需要访问多个微服务，带来分布式事务问题。 没有银弹——不同场景选不同模式：强一致选XA/2PC，高并发选BASE，长事务选SAGA， 想零侵入选SEATA AT。
🏦 XA → 银行转账（强一致） 🛒 BASE → 电商订单（高并发） ✈️ SAGA → 旅行预订（机票+酒店+租车） ⚡ SEATA AT → 通用业务（零侵入）

✅ 优点

  • 不同场景可选择不同模式，灵活度高
  • SEATA AT模式无代码侵入，开发零工作量
  • XA提供强一致性保障
  • BASE可支撑高并发场景

❌ 缺点

  • XA性能差（两次网络交互）
  • TCC和SAGA开发维护成本高
  • BASE通用性有限（只能接受最终一致性）
  • SEATA AT存在使用场景限制
