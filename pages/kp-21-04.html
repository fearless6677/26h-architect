// OSI安全架构 —— 访问控制框架核心逻辑

class AEF {  // 管制执行设备（执行端）
    requestAccess(user, resource) {
        let req = { user, resource, timestamp }
        return ADF.decide(req)   // 向ADF发起访问请求
    }
}

class ADF {  // 决策执行设备（决策端）
    decide(req) {
        let adi = ADI.collect(req) // 收集决策信息
        // ADI = 安全标签 + 用户身份 + 主机IP + 历史决策
        if (policy.evaluate(req, adi))
            return "ALLOW"
        else
            return "DENY"
    }
}

// 抗抵赖 —— 数字签名 5 阶段
function nonRepudiation(contract) {
    evidence = generate(contract)      // ① 生成证据(数字签名)
    transmit(evidence, parties)        // ② 传输证据给各方
    storeAndAck(evidence, TTP)         // ③ 第三方存储+回复确认
    verify(evidence)                   // ④ 验证签名有效性
    resolve(evidence)                  // ⑤ 发生纠纷时仲裁解决
}
OSI安全架构定义了6大安全框架，覆盖安全服务的所有关键方面。核心思路是AEF与ADF分离——决策与执行解耦。
1认证框架
5种鉴别方式：
• 已知的 → 口令/密码
• 拥有的 → IC卡/令牌/USB Key
• 不可变特征 → 指纹/虹膜/人脸
• 受信第三方 → CA证书
• 环境 → 主机IP/MAC地址
多因子认证 = 组合多种方式
2访问控制框架
AEF(管制设备) → 通知 → ADF(决策设备)
ADF 结合 ADI(决策信息:安全标签/用户身份/IP/历史决策) → 做出 允许/禁止 判决
AEF执行 · ADF决策 · ADI信息
3机密性框架
两种手段提供机密性：
• 禁止访问 → 访问控制(路由控制/数据保密)
• 加密 → 对称/非对称加密变换数据
禁止访问 or 加密
4完整性框架
保护数据不被未授权：
• 创建/修改 → 写操作保护
• 删除 → 防删除机制
• 重放 → 序列号/时间戳检测
防创建·防删除·防重放
5抗抵赖框架
5个阶段完整流程：
① 证据生成 → ② 证据传输 → ③ 存储回复 → ④ 证据验证 → ⑤ 解决纠纷
依赖TTP(可信第三方)
生成→传输→存储→验证→纠纷
+深度防御架构(支撑)
三层防御分布：
• 多点防御：边界+环境
• 分层防御：嵌套FW+IDS
• 基础设施：PKI+检测响应
多点+分层+基础设施
核心：OSI安全架构通过深度防御将安全能力分布到整个信息系统，用6大框架覆盖所有安全服务。 关键设计是AEF与ADF分离——管制(执行)与决策解耦，ADI提供决策上下文信息。
🏦 银行网银双因子认证(口令+USB Key) 📜 电子合同数字签名(抗抵赖) 🔐 PKI公钥基础设施 🛡️ 嵌套防火墙+IDS分层防御

✅ 优点

  • 深度防御将安全能力分布至整个信息系统
  • 六大框架覆盖安全服务的所有关键方面
  • AEF与ADF分离实现了决策与执行的解耦

❌ 缺点

  • AEF与ADF分离增加了系统复杂度
  • 分层防御可能导致性能开销
  • 抗抵赖服务依赖第三方证据，实施成本高
