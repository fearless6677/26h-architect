SOA定义与核心概念

SOA是通过中立接口和契约将应用功能单元（服务）联系起来的组件模型

// SOA 核心协议三剑客

// 1. UDDI —— 服务发现（像"黄页电话簿"）
UDDI Registry {
    "查找服务": 查询("财务服务") → "地址: esb.finance.com/ws"
}

// 2. WSDL —— 服务描述（像"菜单"）
WSDL 描述服务的 3个属性：
    做什么    → 接口操作（如：转账、查询余额）
    如何访问  → 协议绑定（SOAP over HTTP）
    位于何处  → 端点地址（URL）

// 3. SOAP —— 消息交换（像"信封"）
SOAP Envelope {
    Header: { 认证信息, 事务ID }
    Body:   {
        <SOAP-ENV:Envelope>
          <SOAP-ENV:Body>
            <Finance:Transfer>
              <From>A001</From>
              <To>B002</To>
              <Amount>1000</Amount>
            </Finance:Transfer>
          </SOAP-ENV:Body>
        </SOAP-ENV:Envelope>
    }
}
核心：SOA是一个组件模型，将应用程序的不同功能单元（称为服务）通过定义良好的接口和契约联系起来。接口采用中立方式定义，独立于硬件平台、操作系统和编程语言。
📌 例子：财务系统通过开放接口向企业提供财务处理服务；WSDL描述服务的3个属性：做什么、如何访问、位于何处。
📒 UDDI = 服务发现（黄页） 📄 WSDL = 服务描述（菜单） 📨 SOAP = 消息交换（信封） 🔗 ESB = 服务总线（中间人）

✅ 优点

  • 打破信息孤岛，实现资源共享
  • 平台/语言无关的互操作性
  • 服务可复用、松耦合

❌ 缺点

  • XML消息格式较冗余，性能开销大
  • 相比微服务，粒度较粗，灵活性不足
